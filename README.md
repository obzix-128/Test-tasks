<details>
<summary><b>Показать условие задания</b></summary>

Необходимо разработать, реализовать и протестировать модуль **bingray_gen**, который осуществляет преобразование числа из двоичного кода в код Грея и наоборот.
- Модуль имеет параметр **WIDTH** для указания ширины операндов.
- Модуль имеет следующие порты:

| Название   | Ширина | Направление | Описание                                                                 |
|------------|--------|-------------|--------------------------------------------------------------------------|
| clk_i      | 1      | Input       | Тактовый сигнал                                                          |
| rst_i      | 1      | Input       | Сигнал сброса                                                            |
| is_binnary | 1      | Input       | Формат входного операнда: 0 – код Грея; 1 – двоичный код                    |
| a_i        | WIDTH  | Input       | Шина входного операнда                                                   |
| result_o   | WIDTH  | Output      | Шина выходного результата                                                |

- Модуль включает регистры (D-триггер) для сохранения результатов операции: полученные в текущий такт, сохранены в регистрах на следующий такт.
  - Шина **result_o** отображает значение, записанное в регистр.
  - Регистры сбрасываемые, значения при сбросе: 0.
- Модуль тактируется сигналом **clk_i**, тактирование по положительному фронту.
- Сброс асинхронный, по активному высокому уровню сигнала **rst_i**.
- Если флаг входного операнда **is_binnary** активен, то число на входе в двоичном формате, если нет, то в формате кода Грея.

**Примеры:**
- WIDTH = 4, is_binnary = 1, a_i = 4’b1001 → result_o = 4’b1101
- WIDTH = 6, is_binnary = 1, a_i = 6’b110001 → result_o = 6’b101001
- WIDTH = 5, is_binnary = 0, a_i = 5’b11110 → result_o = 5’b10100

##### 4. «Поиск первой единицы»
**Статус:** Выполнено :white_check_mark: | **Файлы:** `search_first_one.sv`, `search_first_one_tb.sv`

<details>
<summary><b>Показать условие задания</b></summary>

Необходимо разработать, имплементировать и протестировать модуль `search_first_one`. Модуль выполняет поиск индекса первой единичной `1` во входной шине и записывает результат в регистр.

**Параметры**
Модуль должен поддерживать следующие параметры для конфигурации:

| Параметр | Тип | Описание |
| :--- | :--- | :--- |
| `WIDTH` | `int` | Ширина входной шины данных `data_bus_i`. |
| `R_WIDTH` | `int` | Ширина шины результата `result_index_o`. <br> *Зависимость:* `R_WIDTH = $clog2(WIDTH + 1)` (необходимо для возможности записи значения `WIDTH`, если единицы не найдены). |
| `DIR` | `int` | Направление поиска: <br> `0` — от LSB к MSB (от младшего бита к старшему). <br> `1` — от MSB к LSB (от старшего бита к младшему). |

**Порты**
Модуль имеет следующие интерфейсы:

| Название | Ширина | Направление | Описание |
| :--- | :--- | :--- | :--- |
| `clk_i` | 1 | Input | Тактовый сигнал (положительный фронт). |
| `resetn_i` | 1 | Input | Сигнал сброса (синхронный, активный низкий уровень). |
| `data_bus_i` | `WIDTH` | Input | Шина входных данных. Обновляется каждый такт. |
| `result_index_o` | `R_WIDTH` | Output | Шина индекса результата (значение из регистра результата). |

**Функциональное описание**
Результат вычисляется для массива данных, поступающего на вход `data_bus_i` в текущем такте, и сохраняется в регистр на следующий такт (передний фронт `clk_i`). Выход `result_index_o` отображает значение, хранящееся в регистре. При активном низком уровне сигнала `resetn_i` (`resetn_i == 0`) содержимое регистра сбрасывается в `0`. Сброс синхронный.

В зависимости от значения параметра `DIR`, модуль должен определять индекс первого найденного бита, равного `1`:
1.  **DIR = 0 (Поиск от LSB):** Поиск начинается с младшего бита (индекс 0). Возвращается индекс самого младшего установленного бита.
2.  **DIR = 1 (Поиск от MSB):** Поиск начинается со старшего бита (индекс `WIDTH-1`). Возвращается индекс самого старшего установленного бита.
3.  **Единицы не найдены:** Если во входной шине `data_bus_i` все биты равны `0`, в регистр записывается значение, равное ширине шины (`WIDTH`). Это значение на единицу больше максимального возможного индекса бита.

</details>

##### 4. Crossbar
**Статус:** Выполнено :white_check_mark: | **Файлы:** `crossbar.sv`, `crossbar_tb.sv`

<details>
<summary><b>Показать условие задания</b></summary>

Необходимо разработать, реализовать и протестировать модуль приоритетного коммутатора (crossbar) на языке SystemVerilog или Verilog. Модуль должен осуществлять коммутацию данных с входных портов на выходные порты, на основе адресов, указанных на входных портах. При получении одинаковых валидных адресов, коммутатор должен выдать приоритет коммутации входному порту с бо́льшим индексом.
- Модуль параметризуемый, имеет следующие параметры:

| Название    | Описание                                                                 | Значение по умолчанию |
|-------------|--------------------------------------------------------------------------|-----------------------|
| INPORT_NUM  | Количество входных портов коммутатора                                    | 4                     |
| ADDR_W      | Ширина, необходимая для адресации всех выходных портов                   | 3                     |
| DATA_W      | Ширина коммутируемых данных                                            | 8                     |
| OUTPORT_NUM | Количество выходных портов коммутатора                                   | 6                     |

- Модуль имеет следующие порты:

| Название  | Размерность           | Направление | Описание                                                                 |
|-----------|-----------------------|-------------|--------------------------------------------------------------------------|
| addrin_i  | INPORT_NUM x ADDR_W   | Input       | Массив входных адресов. **addrin_i[i]** – адрес **i-го** входного порта, указывающий номер на какой выходной порт необходимо скоммутировать данные **datain_i[i]** |
| datain_i  | INPORT_NUM x DATA_W   | Input       | Массив входных данных. **datain_i[i]** – данные на **i-ом** входном порту |
| validin_i | INPORT_NUM            | Input       | Массив валидности входных данных и адресов. **validin_i[i]** == 1’b1 – данные и адрес на **i-ом** входном порту валидны |
| dataout_o | OUTPORT_NUM x DATA_W  | Output      | Массив выходных данных. **dataout_o[j]** – данные на **j-ом** выходном порту |

- Модуль **crossbar** представляет собой комбинационную схему (без триггеров).
- Коммутация данных с **i-го** входного порта **datain_i[i]** осуществляется только при наличии соответствующей валидности **validin_i[i]**.
- Значение по умолчанию на выходных портах – ноль. Если для конкретного выхода не пришел запрос на коммутацию, то на данном выходном порту должно быть нулевое значение.
- Разрешение конфликтов – приоритет старшему входному порту. Возможна ситуация, когда два или более входных портов одновременно посылают данные на один выходной порт, в таком случае входной порт с бо́льшим индексом имеет приоритет над входным портом с меньшим индексом.
- При отсутствии конфликтов по адресам входные порты коммутируют свои данные в соответствующие выходные порты.
- Если количество выходных портов не кратно степени 2, то в случае поступления валидного адреса **addrin_i[i]**, значение которого превышает число выходных портов, коммутатор не должен произвести коммутацию.

</details>

- Если один из операндов **qNaN** – результат **qNaN**.
- При переполнении формата FP32, результат – максимально возможное положительное/отрицательное число в формате FP32 (не бесконечность).

<details>
<summary><b>Показать условие задания</b></summary>

Необходимо разработать, реализовать и протестировать модуль ROB (reordering buffer), осуществляющий буферизацию транзакций между master и slave модулями и восстановление исходного порядка перед возвращением в master.
Требования к модулю: 
- Модуль имеет следующие параметры: 

| Параметр   | Описание                                                                           |
|------------|------------------------------------------------------------------------------------|
| INFO_WIDTH | Ширина шины служебных данных запросов к slave                                      |
| DATA_WIDTH | Ширина шины данных ответов от slave                                                |
| ID_WIDTH   | Ширина идентификатора транзакции                                                 |
| BUF_SIZE   | Размер буфера                                                                      |

- Модуль содержит буфер транзакций глубиной **BUF_SIZE**; 
- Модуль принимает транзакции с служебной информацией запросов к slave от master, присваивает им идентификатор и отправляет их на обработку в сторону slave с задержкой в 1 такт; 
- Модуль принимает обработанные транзакции от slave и возвращает их в сторону master в том же порядке, в котором он получал их от master, с задержкой в 1 такт или более; 
- Slave может отвечать обработанными транзакциями в порядке, отличном от того, в котором они приходили, но при сохранении значения **id** каждой транзакции; 
- Транзакции считаются отправляемыми по шине только при сигнале **vld** соответствующей шины в 1; 
- Считается, что master всегда готов принимать обработанные транзакции; 
- Транзакция может читаться отправленной в slave только когда происходит хендшейк на интерфейсе отправки запросов в slave, т.е. когда **slave_vld_o** и **slave_rdy_i** выставлены в 1 одновременно; 
- При заполнении внутреннего буфера модуля флаг **buf_full_o** выставляется в 1; 
- Попытки отправить новую транзакцию от master через заполненный ROB игнорируются и не меняют состояние модуля.

</details>

##### 6. Частично ассоциативный массив
**Статус:** Выполнено :white_check_mark: | **Файлы:** `set_associative.sv`, `set_associative_tb.sv`

<details>
<summary><b>Показать условие задания</b></summary>

Вам необходимо разработать, имплементировать и протестировать модуль **set_associative**, являющийся частично ассоциативным массивом.

**Параметры**

| Название | Описание |
| :--- | :--- |
| `BLK_WIDTH` | Ширина одного блок данных в байтах |
| `WAY_CNT` | Степень ассоциативности |
| `ADDR_WIDTH` | Ширина адреса |

**Описание работы**
* Массив состоит из наборов (set), которые состоят из `WAY_CNT` блоков данных. У каждого блока данных имеется флаг валидности.
* Адреса чтения и записи состоят из полей `tag` (старшие биты) и `set_index` (младшие биты).
* Флаги записи и чтения могут быть выставлены одновременно.

**Запись в массив:**
* При выставлении сигнала `wr_en_i[i]` на шину `data_i` подаются данные для записи в блок данных с индексом `i`.
* Набор определяется с помощью адреса `wr_addr_i`.
* С полученными данными записывается `tag` и выставляется флаг валидности в соответствующем блоке данных. Запись занимает 1 такт.

**Чтение из массива:**
* При выставлении сигнала `read_en_i` в следующем такте на шине `data_o` ожидаются прочитанные данные по адресу `read_addr_i`. Также на шине `hit_miss_o` ожидается значение валидности прочитанных данных.
* Прочитанные данные считаются невалидными, если их флаг валидности равен 0 или по запрошенному адресу не найдены данные (не найден запрошенный `tag`).

**Сброс:** Синхронный, по активному низкому уровню сигнала `resetn_i`. После сброса данные в массиве становятся невалидными (без перезаписи массивов нулями).

</details>

#### Задача 1: Верификация сумматора
 
Для проверки корректности работы устройства сложения двух целых знаковых чисел фиксированной разрядности выбраны 5 комбинаций, покрывающих основные классы эквивалентности и граничные значения.
 
**Таблица 1 — Выбранные тестовые комбинации для сумматора**
 
| № | A | B | Ожидаемый результат | Обоснование |
|---|---|---|---------------------|-------------|

| 2 | 1 | 1 | 2 | Базовое сложение без переносов. |

| 5 | MAX_POS | MIN_NEG | -1 | Знаковая асимметрия в дополнительном коде. |
 
 
#### Задача 2: Электронный кодовый замок
 
Устройство рассматривается как конечный автомат (FSM). Основные изъяны часто связаны с некорректными переходами и отсутствием сброса буфера.
 
**Таблица 2 — Набор тестов для кодового замка (пароль 1-4-6)**
 
| № | Сценарий | Ожидаемое поведение | Цель теста |
|---|----------|---------------------|------------|
| 1 | 1 → 4 → 6 → ВВОД | Открыт | Happy path (основной успех). |
| 2 | 1 → 2 → 3 → ВВОД | Закрыт | Защита от ложного срабатывания. |
| 3 | 9, 9, 1 → 4 → 6 → ВВОД | Закрыт | Защита от поиска подстроки. |
| 4 | 1 → 4 → 6 (без ВВОД) | Закрыт | Проверка участия кнопки подтверждения. |
| 5 | Ошибка → 1 → 4 → 6 → ВВОД | Открыт | Проверка сброса состояния. |
